# Claude Development Guidelines

## Documentation Standards

- Assume senior engineer audience
- Avoid excessive verbosity in comments and documentation
- Cover relevant information concisely
- Focus on non-obvious aspects and rationale over what is self-evident from code

## Feature Documentation

For each major feature, add a description to `docs/FEATURES.md`.

**Keep documentation current:** When modifying a feature's behavior, update its documentation in the SAME commit:
- Event schemas change? Update FEATURES.md
- API endpoints change? Update FEATURES.md
- Performance characteristics change? Update FEATURES.md
- Configuration options change? Update FEATURES.md

Documentation drift happens when code changes without documentation updates. Prevent this by treating docs as part of the implementation, not separate.

Design documents in `docs/` must be implementation-complete specifications. A new Claude instance should be able to implement the feature from the design doc alone without needing to ask questions or make assumptions.

## Version Control

**Git Worktrees for All Changes:**

Make each change in a git worktree. Create a NEW worktree for each change - do NOT reuse worktrees.

IMPORTANT: Always pull master BEFORE creating a worktree to ensure you're working from the latest code.

```bash
# For each change:
cd /home/claude-9/bells
git pull origin master                                      # ← REQUIRED: Pull first
git worktree add -b descriptive-branch-name ../bells-descriptive-name
cd ../bells-descriptive-name
# ... make changes, commit ...
cd /home/claude-9/bells
git merge descriptive-branch-name --no-edit
git push origin master
```

Benefits:
- User's server keeps running in main directory while you work
- No file changes in user's active directory
- Clean separation of each change
- No need to stash or switch branches

Commit after each change. Use descriptive commit messages but avoid excessive verbosity.

NEVER commit tokens or secrets.

## Code Standards

Write idiomatic Ruby and follow best practices for Ruby applications.

## Security: Glob Usage

AVOID using `Dir.glob`, `File.glob`, or shell glob expansion in lib code:

- ❌ BAD: `Dir.glob("#{user_input}/**/*.rb")` - user input can contain glob metacharacters
- ❌ BAD: `Dir.glob("#{path}/*.txt")` - path might contain `*`, `?`, `[`, `]`, `{`, `}`
- ✅ GOOD: Use `Dir.entries` or `Find.find` with explicit pattern matching
- ✅ PERMITTED in test suite ONLY: `Dir.glob("tmp/test_#{id}/*.part.*")` if you can prove:
  - No possibility of user input interpolation
  - All interpolated values are controlled (hardcoded test IDs, PIDs, etc.)
  - No unintentional special characters in the pattern

Glob metacharacters (`*`, `?`, `[`, `]`, `{`, `}`) in paths can cause:
- Directory traversal if user input contains `../`
- Unintended file matching if paths contain special characters
- Security vulnerabilities if glob results are used in sensitive operations

In lib code, glob is generally not safe because:
- Paths often come from external sources (PRs, artifacts, users)
- Even internal paths might contain unexpected characters
- Explicit iteration with pattern matching is safer and more predictable

## Concurrency and Threading

NEVER do network I/O (or any blocking operation) while holding a lock:

- ❌ BAD: `@mutex.synchronize { result = http_get(url) }`
- ✅ GOOD: Read cache under lock → do I/O outside lock → write result under lock
- Locks should only protect memory access, not I/O operations
- Network I/O is unpredictable (50ms-30s) and will block all threads waiting for the lock
- Hold locks for the minimum time necessary (typically <1ms for simple hash operations)

Never hold locks during:
- HTTP requests, API calls
- File I/O (especially reading large files)
- Database queries
- Sleep or wait operations
- Complex computations (>1ms)

## Testing Standards

NEVER use actual GitHub tokens when running tests. Always use mocked data, VCR cassettes, or test fixtures instead of making real API calls with real credentials.

- Use VCR cassettes (`spec/fixtures/cassettes/`) for integration tests
- Mock Octokit client instances for unit tests
- Use WebMock to prevent accidental real HTTP requests
- Test data should be realistic but never contain real credentials or sensitive data

## API Integration

Always use pagination when fetching data from APIs. Never assume results fit in a single page unless the API guarantees it or you're intentionally limiting results for UX reasons (e.g., homepage display).

## Correctness and API Pagination

**Behavior must be correct at all times.** Performance optimizations cannot result in incorrect behavior.

**CI Job Counts and Data Accuracy:**
- Incorrect CI job counts are NOT acceptable
- Missing CI jobs are NOT acceptable
- Displaying partial data (e.g., "98 passed" when there are actually 456) is incorrect behavior
- All counts shown to users must accurately reflect the complete data set

**GitHub API Pagination Requirements:**
- ALL GitHub API calls must be checked for proper pagination usage
- By default, fetch the complete result set using pagination or `auto_paginate`
- Any API calls that intentionally retrieve partial data (e.g., `per_page: 100` without pagination) must be:
  1. Critically evaluated for correctness impact
  2. Clearly documented with code comments explaining:
     - WHY partial data is acceptable
     - WHAT the tradeoffs are (performance vs accuracy)
     - HOW this affects user-visible behavior
     - WHAT edge cases might be missed

**Example of Acceptable Partial Data Retrieval:**
```ruby
# ACCEPTABLE: Limit for performance when partial data doesn't affect correctness
def recent_commits(limit: 20)
  # Only fetch most recent commits for display purposes
  # Users understand this is a "recent" view, not complete history
  @client.commits(REPO, per_page: limit)
end
```

**Example of Unacceptable Partial Data Retrieval:**
```ruby
# UNACCEPTABLE: Shows incorrect counts to users
def check_runs_for_pr(pr_number, limit: 100)
  # ❌ BAD: Returns only 100 runs, but user sees "98 passed"
  # ❌ Actual count might be 456 passed - this is incorrect behavior
  # ❌ Missing 358 jobs that user expects to see
  response = @client.check_runs_for_ref(REPO, sha, per_page: limit)
  response[:check_runs]
end
```

**When Performance Optimization is Needed:**
- Use proper pagination to fetch complete data
- Cache the complete result set
- Use background jobs to pre-fetch data
- NEVER sacrifice correctness for performance

## Error Handling

Always report errors and warnings in the UI where relevant. Users should see what went wrong, not just silent failures in logs.

## Infrastructure Failure Detection

When modifying infrastructure failure detection code, update `docs/implementing-infrastructure-failure-detection.md` to reflect the changes:
- When adding/removing patterns in `INFRASTRUCTURE_PATTERNS`, update the pattern list in the doc
- When changing the categorization logic in `check_for_infrastructure_failure()`, update implementation steps
- When modifying error snippet extraction in `extract_error_snippet()`, update the implementation details

This document is used by other Claude instances to implement infrastructure detection in related tools.

## Testing and Development Servers

When starting web servers for testing or investigation purposes:
- NEVER use the default port (9292 for this app, 3000 for Rails, etc.)
- Always use a non-default port to avoid conflicts with existing development servers
- Example: Use ports 8765, 8888, 9999, or other non-standard ports
- Reason: User may already have development server running on default port

## CLI Wrapper (bin/bells)

All command-line options in bin/bells MUST have short (single-letter) options in addition to long options:
- Good: `-a, --author AUTHOR` (both short and long)
- Bad: `--author AUTHOR` (long only)
- Reason: Short options provide better UX for frequently used commands

## UI and UX Standards

NEVER add CSS animations, transitions, or JavaScript delays to the UI:
- NO fade-ins, fade-outs, slide animations, or transitions
- NO artificial delays (setTimeout, sleep) to "smooth" UI updates
- NO opacity transitions or transform animations
- Reason: Animations make the UI slower on purpose and degrade user experience

If content loads instantly, show it instantly. Do not artificially slow down the UI.

## Communication with User

When the user says "suggest" or "propose improvements", this means:
- Provide analysis and recommendations in text/markdown
- Do NOT make code changes
- Do NOT implement the suggestions
- Wait for explicit approval (e.g., "go", "implement", "do it") before coding
- Reason: User wants to review proposals before committing to implementation
