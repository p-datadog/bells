# Claude Development Guidelines

## Documentation Standards

- Assume senior engineer audience
- Avoid excessive verbosity in comments and documentation
- Cover relevant information concisely
- Focus on non-obvious aspects and rationale over what is self-evident from code

## Feature Documentation

For each major feature, add a description to `docs/FEATURES.md`.

Design documents in `docs/` must be implementation-complete specifications. A new Claude instance should be able to implement the feature from the design doc alone without needing to ask questions or make assumptions.

## Version Control

Commit after each change. Use descriptive commit messages but avoid excessive verbosity.

After committing to a feature branch, merge to master.

NEVER commit tokens or secrets.

## Code Standards

Write idiomatic Ruby and follow best practices for Ruby applications.

## Testing Standards

NEVER use actual GitHub tokens when running tests. Always use mocked data, VCR cassettes, or test fixtures instead of making real API calls with real credentials.

- Use VCR cassettes (`spec/fixtures/cassettes/`) for integration tests
- Mock Octokit client instances for unit tests
- Use WebMock to prevent accidental real HTTP requests
- Test data should be realistic but never contain real credentials or sensitive data

## API Integration

Always use pagination when fetching data from APIs. Never assume results fit in a single page unless the API guarantees it or you're intentionally limiting results for UX reasons (e.g., homepage display).

## Error Handling

Always report errors and warnings in the UI where relevant. Users should see what went wrong, not just silent failures in logs.

## Infrastructure Failure Detection

When modifying infrastructure failure detection code, update `docs/implementing-infrastructure-failure-detection.md` to reflect the changes:
- When adding/removing patterns in `INFRASTRUCTURE_PATTERNS`, update the pattern list in the doc
- When changing the categorization logic in `check_for_infrastructure_failure()`, update implementation steps
- When modifying error snippet extraction in `extract_error_snippet()`, update the implementation details

This document is used by other Claude instances to implement infrastructure detection in related tools.

## Testing and Development Servers

When starting web servers for testing or investigation purposes:
- NEVER use the default port (9292 for this app, 3000 for Rails, etc.)
- Always use a non-default port to avoid conflicts with existing development servers
- Example: Use ports 8765, 8888, 9999, or other non-standard ports
- Reason: User may already have development server running on default port
